# 33 Party 产品功能总文档

更新时间：2026-06-02

本文档用于统一 33 Party 酒吧拼台小程序的产品功能范围，供产品、设计、前端实施和后端联调用作共同基准。本文档不预设视觉风格，不要求采用特定设计语言；设计师可以自由探索视觉方向，只需覆盖本文定义的功能、状态、信息层级和可用性约束。后续如果找设计师重做前端，建议以本文档为页面和状态清单，设计完成后再按设计稿实施。

## 1. 产品定位

33 Party 是面向酒吧、夜店、拼台场景的小程序工具，核心目标是让用户通过扫码或链接进入某个主局/桌房，查看现场信息、成员和聊天内容，并通过管理员完成联系、占位、现场协调和安全处置。

产品不是公开社交广场，也不是用户互加微信平台。当前规则明确限制用户之间直接互加微信，用户只能联系管理员，管理员可联系用户并处理现场事务。

## 2. 用户角色

### 2.1 普通用户

普通用户是扫码/链接进入主局的人。主要行为包括：

- 进入主局或指定桌房。
- 首次进入时完善头像、昵称、性别，阅读并同意协议，确认已满 18 周岁。
- 查看主局状态、开局时间、酒吧地址、桌房信息、成员列表。
- 复制管理员微信，联系管理员处理到店、拼桌、改人数等事项。
- 打开酒吧地图导航。
- 在已占位后参与聊天。
- 发送文字、语音、照片、视频、微信会话图片/GIF 表情。
- 发送闪照。
- 点赞消息。
- 引用消息回复。
- 预览图片、头像、视频。
- 长按消息或成员发起举报。
- 开启微信订阅消息提醒。

### 2.2 管理员/客服/局头

管理员是现场运营和安全管理角色。主要行为包括：

- 登录管理台。
- 查看当前局和所有桌房的状态。
- 创建新局。
- 配置酒吧名称、地址、地图坐标、桌房号、人数、开局时间。
- 编辑对用户可见的管理员微信。
- 查看桌房成员、占位人数、未占位人数、剩余座位、消息数、照片数。
- 复制入局码、入局链接、管理员微信、成员微信。
- 展示/保存小程序码。
- 设置成员为已占位。
- 设置或取消局头。
- 移除成员。
- 解散当前局。
- 多选删除已结束局。
- 查看举报列表。
- 删除违规消息。
- 封禁/解封用户。
- 处理或驳回举报。

### 2.3 后台系统

后台系统负责：

- scene、partyId、tableId、inviteCode 入口解析。
- 用户、管理员、局、桌房、成员、消息、订阅、举报、联系记录的数据管理。
- WebSocket 实时广播。
- 微信订阅消息发送。
- 上传文件存储和访问。
- 闪照/过期内容清理。
- 权限和安全规则校验。

## 3. 信息架构

小程序当前包含三个主要页面：

- 用户房间页：`frontend/pages/room/index`
- 管理员页：`frontend/pages/admin/index`
- 法务/协议页：`frontend/pages/legal/index`

后台接口由 `backend/server.py` 提供，数据存储为 SQLite。

## 4. 用户端功能

### 4.1 入局入口

用户可通过以下方式进入：

- 扫码进入：使用微信小程序 scene。
- 链接进入：携带 `partyId`、`tableId` 或 `inviteCode`。
- 指定桌房进入：携带 `partyId` 和 `tableId`。
- 通过 scene 查询主局和默认桌房。

入口解析规则：

- `scene` 可映射到主局或具体桌房。
- `partyId + tableId` 优先直达指定房间。
- `inviteCode` 可作为 scene 的替代入口。
- 无后端或接口失败时，可回退本地演示数据。

### 4.2 首次资料完善

用户首次进入需要完成资料：

- 选择头像。
- 输入昵称。
- 选择性别：女、男、保密。
- 阅读并同意《用户协议》《隐私政策》《社区规范》。
- 确认本人已满 18 周岁。
- 同步微信登录 openid。
- 保存资料到本地缓存和后端。

校验规则：

- 头像和昵称不能为空。
- 必须同意协议并确认年满 18 周岁。
- 后端会保存协议同意时间和年龄确认时间。
- 用户资料更新会通过 WebSocket 同步到房间内。

### 4.3 房间信息展示

房间页需要展示：

- 主局名称。
- 状态：人数未满、人数已满、已结束。
- 开局时间。
- 管理员/局头名称。
- 管理员微信。
- 酒吧名称。
- 酒吧地址。
- 距离或现场确认提示。
- 桌房号。
- 入局码。
- 占位人数/总人数。
- 剩余座位。
- 成员数量。

状态展示规则：

- 已占位人数小于容量：人数未满。
- 已占位人数等于容量：人数已满。
- 主局或桌房已结束：已结束。
- 用户端不直接暴露数据库 `open` 状态。

### 4.4 管理员联系

用户端只允许联系管理员：

- 展示管理员微信。
- 点击复制管理员微信。
- 后端记录一次 `user -> admin` 联系请求。
- 不展示其他用户微信。
- 不提供用户互加入口。

权限规则：

- `user -> admin`：允许。
- `admin -> user`：允许。
- `user -> user`：禁止，接口返回 403。

### 4.5 地图导航

用户可点击酒吧地址旁的地图入口：

- 使用 `wx.openLocation` 打开微信地图。
- 使用创建局时保存的经纬度。
- 若缺少经纬度，提示暂无可导航坐标。

设计要求：

- 地图入口应是轻量操作，不应占据半行或压缩地址内容。
- 地址长文本需要省略或换行策略，不能撑破布局。

### 4.6 成员列表

用户端成员列表展示：

- 头像。
- 昵称。
- 性别。
- 是否已占位。
- 在线状态。
- 成员数量。

排序规则：

- 已占位成员排在未占位成员前。
- 同状态成员按加入顺序或成员 ID 稳定排序。

交互能力：

- 点击头像可预览头像。
- 长按成员可举报。
- 不展示用户微信号。

### 4.7 入局与占位

用户完善资料后可加入主局。

当前规则：

- 新用户入局默认是未占位 `ghost`。
- 管理员可将用户设置为已占位 `seated`。
- 只有已占位用户才能发言。
- 未占位用户看到“占位后发言”的输入区提示。
- 已结束局不能继续入局、拉房间或发送消息。

### 4.8 聊天能力

聊天支持以下消息类型：

- 文字消息。
- 语音消息。
- 照片消息。
- 视频消息。
- 微信会话图片/GIF 表情。
- 系统消息。
- 闪照。

通用消息能力：

- 当前用户自己的消息靠右展示。
- 其他用户消息靠左展示。
- 展示发送者昵称、头像、时间、点赞数。
- 支持消息点赞。
- 支持长按引用消息。
- 支持长按举报消息。
- 被管理员删除的消息展示为“该消息已被管理员删除”。
- 闪照过期后不再展示原内容。

### 4.9 文字消息

用户可输入文字并发送。

规则：

- 仅已占位用户可发送。
- 发送中需要避免重复点击。
- 发送失败时可本地追加并提示后端暂未接收。
- 支持引用上一条消息。

### 4.10 语音消息

语音流程：

- 长按语音按钮开始录音。
- 松手或点击完成结束录音。
- 可取消录音。
- 录音结束后进入预览/待发送状态。
- 可发送或丢弃语音。
- 可播放语音。
- 播放中显示语音波形状态。

规则：

- 当前环境不支持录音时需要提示。
- 录音过短或未生成文件时需要提示。
- 语音消息展示时应按内容宽度收缩，不能出现大面积空白。

### 4.11 照片和视频

用户可通过微信媒体选择器发送：

- 相册图片。
- 拍摄图片。
- 相册视频。
- 拍摄视频。

规则：

- 照片可普通发送，也可在开启闪照后作为闪照发送。
- 视频使用预览媒体能力。
- 图片使用自定义可缩放预览。
- 上传后保存稳定 `mediaUrl`。

### 4.12 表情

表情能力：

- 文字输入框支持系统 emoji。
- 表情入口通过 `wx.chooseMessageFile` 从微信会话选择图片/GIF。
- 以 `emoji` 消息类型发送。

说明：

- 微信没有开放直接调起用户收藏表情包面板的小程序 API。
- 当前实现是公开 API 能覆盖的最接近微信原生链路。

### 4.13 闪照

用户可开启闪照开关：

- 支持 5 秒、10 秒等时长选项。
- 闪照保存过期时间。
- 倒计时结束后前端隐藏内容。
- 后端支持过期内容清理。

设计要求：

- 闪照开关应轻量，不抢占输入栏主空间。
- 倒计时提示要清楚但不过度打扰。

### 4.14 媒体预览

媒体预览包含：

- 图片预览。
- 头像预览。
- 视频预览。
- 图片支持缩放、拖动、双击缩放。
- 预览层可关闭。

### 4.15 微信订阅消息

用户可开启订阅消息提醒：

- 首次资料面板中可请求订阅。
- 房间页通知入口可请求订阅。
- 后端保存订阅状态。
- 新消息写入后，后端尽力发送微信订阅消息。
- 页面内不做自定义弹窗。

订阅字段：

- 主局/房间。
- 发送人。
- 消息内容。
- 桌号。

### 4.16 举报

用户可举报：

- 消息。
- 成员/用户。

举报原因：

- 骚扰辱骂。
- 色情低俗。
- 诈骗引流。
- 广告刷屏。
- 侵犯隐私。
- 其他。

举报结果：

- 后端生成举报记录。
- 管理员在管理台查看并处理。

### 4.17 法务内容

法务页包含：

- 用户协议。
- 隐私政策。
- 社区规范。

核心规则：

- 服务面向年满 18 周岁的用户。
- 禁止违法违规、色情低俗、暴力恐吓、赌博毒品、诈骗引流、广告刷屏、侵犯隐私、骚扰辱骂、冒充他人等行为。
- 平台可删除内容、移除成员、封禁账号、保存证据并配合监管。

## 5. 管理员端功能

### 5.1 管理员登录与授权

管理员入口支持：

- 管理员 ID。
- 管理密钥 `adminKey`。
- 微信登录 code。
- 管理员 openid 绑定。
- token 登录态。

权限规则：

- 管理端接口需要 `X-Admin-Key` 或 `X-Admin-Token`。
- 管理员只能管理自己名下的局。

### 5.2 管理台总览

管理台总览展示：

- 桌房数。
- 在线成员数。
- 照片数。
- 需关注数量。
- 桌房筛选：全部、未满、已满、关注、结束。
- 当前桌房列表。
- 当前选中桌房详情。

桌房卡片展示：

- 桌房号。
- 主局名称。
- 状态。
- 已占位人数/容量。
- 局头。
- 占位进度。
- 消息数。
- 照片数。
- 未占位人数。
- 成员头像组。

### 5.3 管理员微信管理

管理员可编辑对用户可见的微信号：

- 输入微信号。
- 保存到后端 `admins.wechat_id`。
- 保存后用户端展示新微信。
- 管理台可复制管理员微信。

### 5.4 创建新局

管理员可创建新局，字段包括：

- 局名称。
- 桌/房号。
- 容量人数。
- 开局时间。
- 门店/酒吧名称。
- 酒吧地址。
- 纬度。
- 经度。

创建结果：

- 创建酒吧记录。
- 创建主局记录。
- 创建默认桌房。
- 生成 scene code。
- 生成桌房 share scene/入局码。
- 返回新局和桌房数据。

地图选点：

- 管理员可调用微信地图选点。
- 选点后回填酒吧名称、地址、经纬度。
- 这是用户端准确导航的前置条件。

### 5.5 局生命周期管理

管理员可解散当前局：

- 二次确认。
- 后端将主局和桌房状态置为已结束。
- 在线用户通过 WebSocket 收到 `party.ended`。
- 用户不能继续入局、拉取房间、发送消息。

管理员可删除已结束局：

- 已结束局支持多选。
- 删除时清理桌房、成员、消息、举报、订阅、联系记录等关联数据。
- 未结束局不能删除。

### 5.6 入局码、链接、小程序码

管理台支持：

- 查看入局码。
- 复制入局码。
- 获取入局链接。
- 展示入局信息弹层。
- 生成/下载小程序码。
- 预览小程序码。
- 保存小程序码到相册。
- 复制入局链接。

设计要求：

- 入局码通常较长，不能竖排或撑破布局。
- 复制、展示、保存等操作应是小型操作按钮。
- 小程序码弹层需要清楚展示二维码、scene、错误状态和操作按钮。

### 5.7 成员管理

成员详情展示：

- 头像。
- 昵称。
- 角色：局头或成员。
- 在线状态。
- 占位状态。
- 微信号。
- 封禁状态。

管理员可操作：

- 复制成员微信。
- 联系成员。
- 设置为已占位。
- 设置为局头。
- 取消局头。
- 移除成员。
- 封禁用户。
- 解封用户。

安全设计要求：

- 高风险操作需要二次确认或放入二级 Action Sheet。
- 移除、封禁、解散、删除应使用危险色。
- 现场高频操作应减少误触。

### 5.8 举报管理

管理台支持举报列表：

- 按状态筛选：待处理、已处理、已驳回。
- 展示举报原因。
- 展示桌房号。
- 展示创建时间。
- 展示举报人。
- 展示举报对象：消息或用户。
- 展示被举报消息内容。

举报处理动作：

- 删除被举报消息。
- 封禁被举报用户。
- 解封用户。
- 移除成员。
- 标记举报为已处理。
- 驳回举报。

举报状态：

- `pending`：待处理。
- `resolved`：已处理。
- `rejected`：已驳回。

### 5.9 消息安全处置

管理员可删除违规消息：

- 后端记录删除时间。
- 记录删除人。
- 记录删除原因。
- 房间接口返回“该消息已被管理员删除”。
- 原媒体内容不再对前端展示。

### 5.10 用户封禁

管理员可封禁用户：

- 封禁原因默认为“违规使用”。
- 封禁后用户不能发送消息。
- 封禁后用户不能入局。
- 可解封用户。
- 举报列表和成员列表都可触发封禁/解封。

## 6. 后端和平台能力

### 6.1 接口清单

通用接口：

- `GET /health`
- `GET /api/config`
- `POST /api/uploads`

用户与房间：

- `POST /api/users/login`
- `POST /api/users/profile`
- `GET /api/party/by-scene`
- `POST /api/party/join`
- `GET /api/room`
- `GET /api/messages`
- `POST /api/messages`
- `POST /api/messages/like`
- `POST /api/messages/subscribe`
- `POST /api/photo-burst`
- `POST /api/reports`
- `POST /api/contact/request`

管理员：

- `POST /api/admin/login`
- `POST /api/admin/bind-openid`
- `GET /api/admin/tables`
- `GET /api/admin/tables/invite`
- `GET /api/admin/tables/qrcode`
- `POST /api/admin/profile`
- `POST /api/admin/parties`
- `POST /api/admin/parties/end`
- `POST /api/admin/parties/delete`
- `POST /api/admin/members/seat`
- `POST /api/admin/tables/head`
- `POST /api/admin/members/kick`
- `GET /api/admin/reports`
- `POST /api/admin/reports/resolve`
- `POST /api/admin/messages/delete`
- `POST /api/admin/users/ban`
- `POST /api/admin/users/unban`
- `POST /api/admin/cleanup`

实时连接：

- `GET /ws/room?partyId=...&tableId=...&userId=...`

### 6.2 WebSocket 事件

房间内实时事件包括：

- `connected`：连接成功。
- `message.created`：新消息。
- `message.updated`：消息更新，例如点赞、删除。
- `user.profile.updated`：用户资料更新。
- `member.updated`：成员状态变化，例如占位。
- `party.ended`：主局结束。
- `ping`：心跳。

### 6.3 上传与媒体

后端支持上传：

- 头像。
- 照片。
- 视频。
- 语音。
- 表情图片/GIF。

上传后返回稳定 `mediaUrl`，消息保存媒体 URL。

### 6.4 过期内容清理

系统支持：

- 闪照过期清理。
- 孤儿上传文件清理。
- 管理员手动清理接口。

### 6.5 数据模型

核心表：

- `bars`：酒吧信息和坐标。
- `admins`：管理员资料和微信号。
- `parties`：主局。
- `party_tables`：桌房。
- `users`：用户资料、协议、封禁状态。
- `party_members`：入局关系、角色、占位状态。
- `messages`：消息、媒体、引用、点赞、闪照、删除状态。
- `contact_requests`：微信联系请求记录。
- `message_subscriptions`：微信订阅消息授权记录。
- `reports`：举报记录。
- `admin_sessions`：管理员登录态。

## 7. 核心业务规则

### 7.1 年龄与协议

- 用户必须确认已满 18 周岁。
- 用户必须同意用户协议、隐私政策和社区规范。
- 未完成协议和年龄确认的用户不能正常使用核心功能。

### 7.2 聊天权限

- 用户必须已加入房间。
- 用户必须是已占位 `seated` 状态。
- 被封禁用户不能发送消息。
- 已结束局不能发送消息。

### 7.3 联系权限

- 用户只能联系管理员。
- 管理员可以联系用户。
- 用户之间互相联系禁止。

### 7.4 占位和容量

- 新用户默认未占位。
- 管理员负责占位确认。
- 已占位人数决定“人数未满/人数已满”。
- 总入局人数和未占位人数都需要在管理台可见。

### 7.5 结束和删除

- 局结束后不可继续入局和聊天。
- 只有已结束局可以删除。
- 删除局会清理关联桌房、成员、消息、举报、订阅、联系记录。

### 7.6 内容安全

- 用户可举报消息和用户。
- 管理员可删除消息、移除成员、封禁用户。
- 被删除消息不展示原内容和媒体。
- 被封禁用户无法继续使用关键互动功能。

## 9. 设计约束和注意事项

### 9.1 小程序环境

- 需要适配微信小程序原生组件限制。
- 原生 `button` 有默认最小尺寸和默认样式，需要谨慎使用。
- 高密度页面尽量使用 `view` 做轻量触控按钮。
- 所有长文本必须考虑省略、换行或固定宽度。
- 入局码、微信号、地址、昵称都可能变长。

### 9.2 夜场场景

- 现场使用时信息需要快速扫读。
- 管理端比用户端更强调密度和效率。
- 用户端需要安全、克制、可信，不应像陌生人社交广场。
- 危险操作需要明显但不误触。

### 9.3 视觉方向

本文档不限定 iOS、暗色、赛博、极简、夜店视觉或任何特定风格。设计师可根据品牌定位、目标人群和现场使用场景自行提出视觉方向，但需要满足以下体验原则：

- 信息优先级清晰：主局信息、桌房状态、成员状态、聊天内容和关键操作不能互相抢层级。
- 操作尺寸克制：地图、复制、保存、创建、展示等高频操作不能占据过多内容空间。
- 按钮需要分级：主操作、次操作、危险操作、状态提示应有明确区分。
- 聊天内容按消息本身收缩或排布，避免语音、短文字等内容出现不必要的大面积空白。
- 输入栏需要稳定可用，不应被工具按钮、闪照开关或系统安全区挤压变形。
- 管理端需要支持高密度扫读，用户端需要安全、可信、易理解。
- 小屏状态必须单独检查，不能出现横向溢出、内容遮挡、按钮越界或长文本竖排。

## 11. 后续实施建议

建议下一步流程：

1. 产品确认本文档的功能范围。
2. 设计师按第 8 节输出用户端和管理端设计稿。
3. 设计稿需包含主要页面、弹层、空状态、错误状态、长文本状态、小屏状态。

5. 前端实施时优先重构组件结构，避免继续在旧样式上局部修补。
6. 每完成一个页面，用微信开发者工具检查主流小屏和常规屏幕，确认没有溢出、遮挡、错位、长文本竖排或按钮挤压。
